# Burial Protocol - Game Design Document

기준일: `2026-06-05`
기준 브랜치: `main`
엔진: Godot `4.6`
언어: GDScript

---

## 1. 게임 한 줄 정의

**Burial Protocol**은 좁은 수직 샤프트 안에서 위에서 떨어지는 블록을 파괴하거나 피하고, 쌓이는 모래와 중량 압박을 버티며, Day마다 상점에서 무기, 프로토콜, 모듈, 아이템을 강화해 Day 30까지 생존하는 2D 사이드뷰 생존 로그라이트 게임이다.

---

## 2. 핵심 플레이 감성

Burial Protocol의 핵심 감성은 아래 세 가지다.

1. 위에서 떨어지는 블록을 빠르게 판단하고 처리한다.
2. 처리하지 못한 블록은 모래로 분해되어 전장을 점점 막는다.
3. Day 사이 상점에서 장비와 성장 수단을 구매해 다음 Day를 버틴다.

이 게임은 단순 낙하물 회피 게임이 아니라 **낙하 블록 처리 + 모래 압박 + Day별 장비 성장**이 결합된 세로형 생존 루프다.

장비 성장은 아래 네 축으로 나뉜다.

```text
직접 조작 공격 = 무기
자동 공격 보조 = 프로토콜
빌드 규칙 변형 = 모듈
누적 스탯 성장 = 아이템
```

---

## 3. 현재 구현 상태 요약

현재 프로젝트는 플레이 가능한 코어 루프를 가진 상태다.

구현 완료 또는 실제 연결된 영역:

- Title -> MainHub -> Main -> Result 흐름
- 캐릭터 선택 placeholder와 기본 일꾼 선택
- 난이도 선택과 순차 해금 구조
- 30 Day 런 구조
- 낙하 블록 스폰
- 보스 Day 스폰
- 블록 파괴/분해
- 모래 생성과 모래 시뮬레이션
- 중량 한도 기반 실패
- 좌우 벽 채굴
- 마우스 방향 기반 채굴
- 점프, 추가 점프, 대시, 벽타기
- legacy attack_module 기반 공격 처리
- Day 종료 후 intermission
- 낙하형 키오스크
- Day 상점 UI
- 상점 아이템 5개 롤
- legacy attack_module/function_module/enhance_module 구매 처리
- XP와 레벨업 카드
- 레벨업 카드 5장 제시 방식
- 근거리/원거리 공격력 분리 (`run_bonus_melee_attack_damage`, `run_bonus_ranged_attack_damage`)
- 공격 스타일/연출 스타일 처리 (`AttackModuleStyleResolver`)
- 상점 아이템 랭크별 가격 티어링 (`SHOP_ITEM_RANK_FALLBACK_PRICES`)
- 런타임 스탯 증가
- HUD와 ESC 스탯 패널
- 저장 파일과 최고 기록 저장

중요:

- 위 구현 목록의 `legacy attack_module`은 현재 코드 호환명이다.
- 신규 설계 기준은 `무기 / 프로토콜 / 모듈 / 아이템` 4분류다.
- 문서와 후속 작업에서는 legacy 명칭을 최종 설계처럼 사용하지 않는다.

아직 완성 시스템으로 보지 않는 영역:

- 설정 메뉴 실기능
- 메타 성장 실효과
- 업적 실효과
- 인벤토리/도감성 UI
- 블록 특수 결과의 모든 효과
- 최종 상점 UX/비주얼/한글화
- Weapon / Protocol / Module / Item 데이터 완전 분리
- 프로토콜 드론 장착 UI
- 모듈 패시브 슬롯 UI
- 아이템 소지 제한/unique 처리
- 최종 아트 적용
- 최종 밸런스

## 6. 플레이어

### 6-1. 역할

플레이어는 좁은 중앙 샤프트 안에서 블록을 공격하고, 벽과 모래를 채굴하며, 쌓이는 모래를 피해 생존하는 주체다.

전투 능력은 아래 요소들의 합으로 구성된다.

- 좌/우 슬롯에 장착한 무기
- 드론에 장착한 프로토콜
- 패시브 모듈
- 누적 아이템과 레벨업 카드 스탯

플레이어는 직접 전투와 장비 빌드를 연결하는 플랫폼이며, 전투의 핵심 조작 감각은 무기가 담당한다.

### 6-2. 이동

현재 이동 시스템:

- 좌우 이동
- 공중 이동
- 모래 속 감속
- 점프
- 추가 점프
- 코요테 타임
- 점프 버퍼
- 빠른 낙하
- 좌/우/하 대시
- 배터리 기반 벽타기

주요 수치:

| 항목 | 값 |
|---|---:|
| 지상 이동속도 | `426 px/s` |
| 공중 이동속도 | `373 px/s` |
| 모래 속 이동 배율 | `0.62` |
| 중력 | `2000 px/s^2` |
| 점프 속도 | `-853` |
| 추가 점프 | `1회` |
| 벽 점프 가로 속도 | `480` |
| 빠른 낙하 가속 | `2933 px/s^2` |
| 빠른 낙하 최대 속도 | `1306 px/s` |
| 대시 거리 | `4U` |
| 대시 지속시간 | `0.08초` |
| 대시 쿨다운 | `0.45초` |
| 벽타기 배터리 | `100` |
| 벽타기 소모 | `5/sec` |
| 기본 배터리 회복 | `5/sec` |

### 6-3. 벽타기

벽타기는 고정벽 전용 이동 시스템이다.

발동 조건:

- 대시 중이 아님
- 배터리 `> 0`
- 고정벽 접촉
- 접촉한 벽 방향 입력 유지
- 플레이 가능 상단 제한 아래에 있음

벽타기 중에는 낙하 속도가 제한되고 배터리가 소모된다.
벽타기가 아닐 때는 배터리가 회복된다.

## 7. 장비/성장 객체 체계

Burial Protocol의 장비/성장 객체는 아래 4계층으로 구분한다.

| 계층 | 슬롯/소지 | 역할 |
|---|---|---|
| 무기 (`Weapon`) | 좌/우 슬롯 | 기본공격, 직접 조작 공격 |
| 프로토콜 (`Protocol`) | 드론에 최대 5개 | 자동공격 |
| 모듈 (`Module`) | 최대 5개 | 패시브 스킬, 빌드 변형 |
| 아이템 (`Item`) | 기본 소지 제한 없음 | 스탯 제공, 누적 성장 |

### 7-1. 무기

무기는 좌/우 슬롯에 장착해 플레이어 기본공격의 역할을 한다.

원칙:

- 무기는 입력 기반 공격의 주체다.
- 좌/우 슬롯에 각각 하나씩 장착할 수 있다.
- 무기별로 속성, 유형, 발동 방식, 판정 방식, 연출 스타일을 가진다.
- `melee`, `ranged`는 더 이상 전체 장비 분류가 아니라 무기의 공격 방식 또는 태그로 취급한다.

### 7-2. 프로토콜

프로토콜은 드론에 장착하는 자동공격 루틴이다.

원칙:

- 프로토콜은 드론에 최대 5개 장착한다.
- 프로토콜은 공격키와 무관하게 자동으로 발동한다.
- 프로토콜은 기본공격이 아니며 좌/우 무기 슬롯을 차지하지 않는다.
- 기존 `mechanic` 또는 자동공격 계열의 설계 의도는 장기적으로 프로토콜에 흡수한다.

### 7-3. 모듈

모듈은 최대 5개 장착 가능한 패시브 스킬 장비다.

원칙:

- 모듈은 직접 공격을 생성하지 않는다.
- 모듈은 무기, 프로토콜, 채굴, 생존, 경제 규칙을 변형한다.
- 조건부 효과, 시너지, 특수 룰 변경은 모듈에 우선 배치한다.

### 7-4. 아이템

아이템은 기본적으로 소지 제한 없이 누적되는 스탯 제공 객체다.

원칙:

- 아이템은 장착 슬롯을 차지하지 않는다.
- 아이템은 스탯 증가, 경제 보너스, 단순 누적 효과를 제공한다.
- 특정 강력한 아이템은 개별 소지 제한을 가질 수 있다.
- 단순 스탯 증가는 모듈보다 아이템에 우선 배치한다.

### 7-5. legacy 공격모듈 처리

현재 코드에는 `attack_module`, `function_module`, `enhance_module`이 남아 있다.
이 표현은 신규 설계의 canonical 분류가 아니라 legacy 구현명이다.

| legacy 구현명 | 신규 설계상 재분류 |
|---|---|
| `attack_module` | 무기 또는 프로토콜 |
| `function_module` | 모듈 또는 특수 아이템 |
| `enhance_module` | 아이템 중심, 일부 모듈 가능 |
| `mechanic` 공격모듈 | 프로토콜 후보 |

문서에서는 `공격모듈`을 무기/프로토콜/모듈의 포괄어처럼 사용하지 않는다.

## 8. 채굴 시스템

채굴은 우클릭 기반 환경 상호작용이다.
전투 장비 체계와 별개의 시스템이다.

현재 채굴 대상:

- 모래 셀
- 좌우 벽 서브셀

현재 채굴 비대상:

- 낙하 블록

주요 수치:

| 항목 | 값 |
|---|---:|
| 기본 채굴 데미지 | `1` |
| 채굴 쿨다운 | `0.15초` |
| 채굴 버퍼 | `0.12초` |
| 채굴 거리 | `0.25U` |
| 채굴 높이 | `1U` |

Day 종료 후 intermission 유예 시간이 지나면 채굴만 정지된다.
플레이어 이동, 자연 모래 반응, 플레이어의 모래 밀림은 유지된다.

## 9. 블록 시스템

### 9-1. 최신 블록 모델

최신 기준 블록은 아래 구조다.

```text
Runtime Block = Material x Size + optional Type
```

- `Material`: 재질/성질
- `Size`: 가로/세로 크기
- `Type`: 선택 affix/modifier

기존 코드 호환용으로 `base`라는 이름이 일부 남아 있을 수 있지만, 문서상 개념은 material/size/type을 기준으로 한다.

### 9-2. 스폰 흐름

현재 라이브 스폰은 v1 조합 roll 방식이다.

1. 현재 난이도와 Day 확인
2. `BlockCatalog`에서 유효한 spawn candidate 수집
3. candidate weight 기반 랜덤 선택
4. optional type 선택
5. `BlockSpawnResolver`가 resolved definition 생성
6. `BlockData`로 변환
7. `FallingBlock` 생성

### 9-3. 최종 HP 공식

```text
final_hp =
  BLOCK_HP_PER_UNIT
  x size_hp_multiplier
  x material_hp_multiplier
  x difficulty_hp_multiplier
  x type_hp_multiplier
  x day_hp_multiplier
```

기본값:

- `BLOCK_HP_PER_UNIT = 10.0`

### 9-4. 최종 보상 공식

```text
final_reward =
  BLOCK_REWARD_PER_UNIT
  x size_reward_multiplier
  x material_reward_multiplier
  x type_reward_multiplier
```

기본값:

- `BLOCK_REWARD_PER_UNIT = 5.0`

### 9-5. 최종 모래량 공식

```text
final_sand_units =
  BLOCK_SAND_UNITS_PER_UNIT
  x size_reward_multiplier
  x type_sand_units_multiplier
```

기본값:

- `BLOCK_SAND_UNITS_PER_UNIT = 36.0`

### 9-6. 블록 종료 방식

| 종료 방식 | 결과 |
|---|---|
| 공격으로 파괴 | 골드 + XP 지급 |
| 지형/모래에 닿아 분해 | 모래 생성 |
| 플레이어 압착 | 플레이어 피해 + 모래 생성 |
| Day 30 보스 파괴 | 런 클리어 |
| Day 30 보스 분해 후 생존 | 런 클리어 |

## 10. 모래 시스템

모래는 블록이 분해될 때 생성되는 주요 위험 자원이다.

역할:

- 전장 공간 점유
- 플레이어 이동 방해
- 채굴 대상 제공
- 중량 실패 조건 누적

중량 기준:

| 항목 | 값 |
|---|---:|
| 내부 실패 기준 | `2400 cells` |
| 표시 환산 | `1 cell = 0.1 KG` |
| HUD 최대 표시 | `240.0 KG` |

현재 모래는 자연 낙하, 대각선 확산, 플레이어 밀림, 점프 공간 확보 등을 처리한다.

## 11. Day / Intermission / Shop 루프

### 11-1. Day Active

Day active 중 동작:

- Day 타이머 감소
- 블록 스폰
- 보스 Day 처리
- 무기 공격
- 프로토콜 자동공격
- 모듈/아이템 패시브 효과 적용
- 채굴
- 모래 시뮬레이션
- HP 재생
- XP/골드 획득
- 실패 조건 체크

현재 구현에서는 위 일부가 legacy attack_module 처리에 묶여 있을 수 있다.
문서상 신규 분류는 무기/프로토콜/모듈/아이템으로 유지한다.

### 11-2. Day 종료

Day 1~29에서 시간이 끝나면 결과 화면으로 가지 않고 intermission으로 진입한다.

흐름:

1. Day active 종료
2. 일반 스폰 중단
3. intermission 진입
4. 마지막 활성 블록 정리 대기
5. `1.25초` 뒤 키오스크 투하
6. `3초` 뒤 채굴 잠금
7. 키오스크 상호작용으로 상점 진입

### 11-3. 키오스크

키오스크는 월드 오브젝트로 낙하한다.

| 항목 | 값 |
|---|---:|
| 상호작용 키 | `E` |
| 상호작용 거리 | `2U` |
| 낙하 속도 | `780 px/s` |
| 투하 지연 | `1.25초` |

### 11-4. 상점

상점은 intermission 중 진입 가능한 실제 구매 UI다.

현재 기능:

- 상점 아이템 5개 롤
- 아이템 목록 표시
- 상세 정보 표시
- 가격 표시
- 구매 가능 여부 표시
- 골드 차감
- 구매 성공 시 현재 상점 목록에서 제거
- 골드 지불 reroll
- 슬롯별 lock
- lock된 슬롯은 reroll/다음 상점 롤에서 같은 슬롯의 한 자리를 차지하며 보존
- `Close`
- `Next Day`

상점의 신규 설계 카테고리:

- `weapon`
- `protocol`
- `module`
- `item`

현재 코드의 legacy 카테고리:

- `attack_module`
- `function_module`
- `enhance_module`

현재 legacy 카테고리는 최종 설계가 아니라 마이그레이션 대상이다.

### 11-5. Next Day

`Next Day` 버튼을 누르면:

1. 페이드 아웃
2. 이자 지급
3. 예약된 벽 복구가 있으면 실행
4. intermission 종료
5. Day 증가
6. 새 Day 시간 설정
7. 스폰 타이머 재시작
8. 페이드 인

기본 규칙:

- 벽은 자동 복구하지 않는다.
- 모래도 기본적으로 유지된다.
- 벽 복구는 특정 효과가 예약했을 때만 실행된다.

## 18. 설계 원칙 요약

- 코어 루프를 깨지 않는다.
- 상점은 현재 구현된 시스템으로 취급한다.
- 공격은 무기와 프로토콜로 분리해 설명한다.
- 모듈은 패시브 스킬과 빌드 변형으로 설명한다.
- 아이템은 기본적으로 소지 제한 없는 스탯 제공 객체로 설명한다.
- legacy attack_module 표현은 현재 코드 호환명으로만 사용한다.
- 블록은 material x size + optional type으로 설명한다.
- 모래는 단순 이펙트가 아니라 실패 조건과 전장 압박의 핵심이다.
- Day 전환에서 벽과 모래는 기본 유지한다.
- 메타 시스템은 placeholder 범위를 명확히 표시한다.
