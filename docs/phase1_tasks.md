# Burial Protocol - Phase 1 Tasks

## 현재 기준선 메모 (2026-04-18)

Phase 1은 아래 Phase 0 기준선 위에서 진행한다.

- 게임은 타이틀 화면에서 시작한다
- 메인 허브, 캐릭터 선택, 결과 화면, placeholder 하위 메뉴가 이미 연결되어 있다
- 단일 프로필 저장 구조가 있다
- 좌클릭 공격 / 우클릭 채굴 입력 분리가 끝난 상태다
- 플레이어 대시는 이미 기본 구현이 들어가 있다
- 남은 기초 안정화 이슈로는 벽 채굴 방향 우선순위와 세로 월드 확장성 점검이 있다

## 0. 목적

이 문서는 Burial Protocol의 Phase 1 구현 작업을 실제 개발 단위로 정리한 체크리스트다.

Phase 0의 목표가
중앙 플레이 영역에서 한 판의 생존 루프를 실제로 성립시키는 것이었다면,
Phase 1의 목표는
이미 성립한 생존 루프 위에
플레이어 기동 확장과 블록 위협 다양화를 추가하여
판단 구조를 더 깊게 만드는 것이다.

이번 Phase 1의 핵심은 아래와 같다.

- 플레이어 대시 시스템 추가
- 블록 베이스 시스템 1차 도입
- 블록별 위협도와 처리 우선순위 차이 명확화
- 파괴 / 분해 / 폭발 / 보상 차이 확장
- 이후 블록 속성 시스템을 붙일 수 있는 데이터 구조 마련

완료 기준:

- 플레이어가 좌 / 우 / 하 방향 대시로 즉시 회피할 수 있다
- 블록이 단순 체력 차이를 넘어서 서로 다른 처리 우선순위를 가진다
- 어떤 블록은 고보상이고, 어떤 블록은 실패 시 더 위험하게 작동한다
- 플레이어가 낙하 블록을 보고 우선 대응 판단을 하게 된다
- 향후 블록 속성 시스템을 붙일 수 있는 구조가 준비된다

---

## 1. Phase 1 전체 범위

- `P1-1` Phase 0 마감 점검 및 데이터 정리
- `P1-2` 플레이어 대시 시스템 추가
- `P1-3` 블록 베이스 시스템 1차 도입
- `P1-4` 블록 베이스별 결과 연출 및 판정 확장
- `P1-5` 블록 데이터 구조 정비 및 스폰 확장
- `P1-6` 밸런스 조정 및 통합 테스트

권장 진행 원칙:

- 이미 성립한 Phase 0 루프를 깨지 않는다
- 플레이어 회피 수단을 먼저 정리한 뒤 블록 위협 다양화를 붙인다
- 블록 속성 시스템은 이번 문서 범위에 포함하지 않는다
- 모래 상태이상 / 전파 / EMP / 오염 같은 고난도 확장은 다음 작업 묶음으로 분리한다
- 물통블록, 흡수블록은 구현 난이도 확인 후 후순위로 둔다

---

## 2. Phase 1의 핵심 플레이 확장 방향

Phase 1에서는 아래 변화가 실제 플레이에서 체감되어야 한다.

1. 플레이어는 즉시 회피용 대시를 사용할 수 있다
2. 낙하 블록은 서로 다른 처리 우선순위를 가진다
3. 블록 파괴 / 분해 / 폭발 / 보상 차이가 실제 판단에 영향을 준다
4. 플레이어는 공격, 대시, 채굴, 위치 조정 사이에서 더 자주 선택을 강요받는다

즉,
Phase 1은 생존 루프를 새로 만드는 단계가 아니라,
이미 성립한 생존 루프의 판단 밀도를 높이는 단계다.

- 기존 기본 블록이 여전히 정상 동작한다
- 최소 메타 허브 루프가 깨지지 않는다
- 블록별 체력 / 보상 / 결과 차이를 데이터로 넣을 준비가 된다
- 플레이어 입력 구조가 대시 추가를 감당할 수 있게 정리된다
- 다음 작업부터 하드코딩 없이 기능을 붙일 수 있다

## 6. P1-2 플레이어 대시 시스템 추가

### 목표

플레이어가 낙하 위협과 누적 모래 압박을 더 적극적으로 회피할 수 있도록
기본 기동 확장으로 대시 시스템을 추가한다.

대시는
같은 방향 입력을 짧은 시간 안에 두 번 누르면
해당 방향으로 빠르게 이동하는 회피 기술이다.

### 기본 규칙

- 같은 방향키를 짧은 시간 내에 두 번 입력하면 대시 발동
- 대시 방향은 입력 방향과 동일
- 지원 방향:
  - 좌
  - 우
  - 하
- 위 대시는 구현하지 않는다
- 위 방향 회피 / 상승은 기존 점프와 공중 추가 점프가 담당한다
- 대시 거리는 기본 4U
- 대시는 매우 빠르게 이동해야 한다
- 대시 중 고정 지형 관통은 허용하지 않는다

### 방향별 의도

#### 좌 / 우 대시

- 낙하 블록 회피
- 벽 접근
- 모래 더미 옆 탈출
- 위치 재조정

#### 하 대시

- 빠른 하강
- 상부 압박 회피
- 착지 위치 강제 조정
- 좁은 공간 탈출 보조

#### 위 대시 제외 이유

- 플레이어는 이미 기본 점프 + 공중 추가 점프를 가진다
- 위 대시까지 추가하면 상향 기동 수단이 과해질 수 있다
- 상향 회피는 점프 계열, 수평 / 하향 급회피는 대시 계열로 역할을 분리한다

### 세부 작업

- [ ] 좌 / 우 / 하 방향별 더블탭 판정 구조 구현
- [ ] 방향별 최근 입력 시간 기록 구조 구현
- [ ] 더블탭 허용 시간 상수 정의
- [ ] 대시 거리 상수 정의
- [ ] 대시 속도 또는 대시 지속시간 상수 정의
- [ ] 좌 / 우 대시 구현
- [ ] 하 대시 구현
- [ ] 위 방향 더블탭은 대시로 연결되지 않도록 처리
- [ ] 대시 중 이동 입력 처리 방식 정의
- [ ] 대시 중 충돌 처리 구현
- [ ] 대시 쿨다운 구현
- [ ] 대시 시각 피드백 추가
- [ ] 대시 중 모래와의 상호작용 조정
- [ ] 대시 중 falling block과의 충돌 규칙 확인
- [ ] 디버그 정보로 대시 상태 확인 가능하게 처리

### 산출물

- 방향키 더블탭 기반 대시 입력 시스템
- 좌 / 우 / 하 3방향 대시 이동
- 대시 쿨다운 및 충돌 처리
- 대시 시각 피드백

### 완료 조건

- 같은 방향키를 짧은 시간 안에 두 번 누르면 좌 / 우 / 하 방향 대시가 발동한다
- 플레이어가 해당 방향으로 약 4U 빠르게 이동한다
- 위 방향 더블탭은 대시를 발동하지 않는다
- 벽과 바닥 등 고정 지형을 뚫지 않는다
- 일반 이동 / 점프 / 공중 추가 점프와 구분되는 회피 수단으로 체감된다
- 모래와 낙하 블록 환경 안에서도 규칙이 일관되게 동작한다

## 7. P1-3 블록 베이스 시스템 1차 도입

### 목표

블록마다 HP 배율, 보상, 결과 차이가 생기도록
블록 베이스 시스템을 데이터 중심으로 도입한다.

### 블록 정의

#### 유리블록

- 반투명 흰색으로 표현
- HP 배율 1배
- 파괴 시 주변에 피해를 준다
- 피해 대상은 캐릭터다

#### 나무블록

- 갈색으로 표현
- HP 배율 1배

#### 바위블록

- 회색으로 표현
- HP 배율 1배

#### 대리석블록

- 흰색으로 표현
- HP 배율 1.5배

#### 황금블록

- 금색으로 표현
- HP 배율 2배 ~ 5배
- 배율에 따라 추가 재화를 지급한다

#### 시멘트블록

- 진한 회색으로 표현
- HP 배율 2배

#### 강철블록

- 검정색으로 표현
- HP 배율 3배

#### 폭탄블록

- 빨간색으로 표현
- HP 배율 1.5배
- 모래화 되지 않는다
- 모래화 대신 폭발한다
- 폭발 시 가로세로 5U 범위에 피해를 준다

### 세부 작업

- [ ] 블록 베이스 ID 추가
- [ ] 블록 베이스별 색상 정의
- [ ] 블록 베이스별 HP 배율 정의
- [ ] 블록 베이스별 분해 / 폭발 / 추가 보상 여부 정의
- [ ] 블록 베이스별 디버그 식별 구조 추가
- [ ] 기존 스폰 시스템이 여러 베이스를 생성할 수 있게 확장

### 산출물

- 블록 베이스 데이터 테이블
- 기본 8종 블록 생성 가능 구조
- 베이스별 차등 결과 처리 기반

### 완료 조건

- 8종 블록이 데이터 기반으로 스폰 가능하다
- 각 블록이 서로 다른 체력 배율을 가진다
- 황금블록은 추가 재화 지급이 가능하다
- 폭탄블록은 모래 대신 폭발 처리된다
- 유리블록은 파괴 시 주변 피해가 가능하다

## 8. P1-4 블록 베이스별 결과 연출 및 판정 확장

### 목표

블록 베이스 차이가 단순 수치가 아니라
플레이어가 즉시 이해 가능한 결과 차이로 보이도록 만든다.

### 세부 작업

- [ ] 유리블록 파괴 시 주변 피해 판정 구현
- [ ] 유리블록 파괴 시 전용 시각 연출 추가
- [ ] 황금블록 파괴 시 추가 재화 지급 연결
- [ ] 황금블록 보상량 시각 피드백 추가
- [ ] 폭탄블록 폭발 판정 구현
- [ ] 폭탄블록 폭발 범위 5U 처리 구현
- [ ] 폭탄블록 전용 폭발 연출 추가
- [ ] 폭탄블록은 모래를 남기지 않도록 분기 처리
- [ ] 일반 블록과 특수 결과 블록의 차이가 명확히 보이게 연출 보강

### 산출물

- 파괴 시 주변 피해 구조
- 추가 재화 지급 구조
- 폭발 처리 구조
- 블록별 결과 연출 차이

### 완료 조건

- 유리블록 파괴 시 캐릭터가 주변 피해를 받을 수 있다
- 황금블록 파괴 시 추가 재화가 지급된다
- 폭탄블록은 분해되지 않고 폭발한다
- 폭탄블록 폭발 범위 피해가 정상 적용된다
- 플레이어가 결과 차이를 시각적으로 이해할 수 있다

## 9. P1-5 블록 데이터 구조 정비 및 스폰 확장

### 목표

향후 블록 속성 시스템을 붙일 수 있도록
블록 데이터를 베이스 + 추가 속성 구조로 정리한다.

### 세부 작업

- [ ] block_base와 block_modifier 개념 분리 설계
- [ ] 현재는 block_base만 실제 사용하게 구성
- [ ] 추후 속성 1개 이상 붙일 수 있는 빈 구조 마련
- [ ] 스폰 테이블에서 베이스별 등장 확률 조절 가능하게 정리
- [ ] 난이도에 따른 블록 베이스 등장 비율 조절 여지 마련
- [ ] 황금블록 HP 배율 범위형 처리 방식 결정
- [ ] 폭탄블록처럼 분해 결과를 덮어쓰는 타입 지원 구조 마련

### 산출물

- 향후 속성 시스템 확장 가능한 데이터 구조
- 난이도 기반 스폰 확장 여지
- 베이스와 속성 분리 가능한 틀

### 완료 조건

- 블록 데이터 구조가 베이스 중심으로 정리된다
- 추후 속성 1개 추가가 가능한 구조가 남아 있다
- 스폰 확률과 등장 난이도 조절이 가능하다

## 10. P1-6 밸런스 조정 및 통합 테스트

### 목표

대시 시스템과 블록 베이스 시스템이
실제 생존 루프 안에서 의미 있게 작동하는지 검증한다.

### 통합 체크리스트

- [ ] 대시가 실제 회피 수단으로 작동한다
- [ ] 대시가 기존 점프 / 더블점프와 역할이 겹치지 않는다
- [ ] 유리블록이 일반 블록과 다르게 위험하게 느껴진다
- [ ] 황금블록이 고위험 고보상으로 느껴진다
- [ ] 강철 / 시멘트 / 대리석이 체력 차이로 분명히 구분된다
- [ ] 폭탄블록을 어떻게 처리할지 플레이어 판단이 생긴다
- [ ] 블록 종류 증가가 루프를 어지럽히지 않는다
- [ ] 기본 공격 / 대시 / 채굴 / 회피 루프가 유지된다
- [ ] 성능 저하 없이 일반 플레이가 가능하다

### 핵심 검증 시나리오

#### 시나리오 A - 대시 회피

- [ ] 상단 낙하 위협을 좌 / 우 대시로 피할 수 있다
- [ ] 하 대시로 빠르게 회피 또는 위치 조정이 가능하다
- [ ] 위 방향 더블탭은 대시로 발동하지 않는다

#### 시나리오 B - 체력 차이 판단

- [ ] 서로 다른 HP 배율 블록이 함께 등장한다
- [ ] 플레이어가 제거 우선순위를 체감하게 된다

#### 시나리오 C - 고보상 판단

- [ ] 황금블록 등장 시 더 위험을 감수하고 처리할 이유가 생긴다

#### 시나리오 D - 특수 결과 판단

- [ ] 유리블록은 근접 처리 시 리스크가 있다
- [ ] 폭탄블록은 처리 방식에 따라 위험 / 기회가 달라진다

#### 시나리오 E - 루프 유지

- [ ] 기동 수단과 블록 종류가 늘어도 공격 / 분해 / 모래 압박 / 생존 루프가 끊기지 않는다

### 완료 조건

- 대시가 실제 플레이 판단에 의미를 가진다
- 블록 베이스 차이가 실제 플레이 판단에 영향을 준다
- 블록 종류 증가가 단순 비주얼 차이에 그치지 않는다
- 이후 속성 시스템을 붙일 준비가 된다
- 생존 루프가 여전히 안정적으로 유지된다

- 속성 시스템은 이번 단계에서 구조만 열고 본 구현은 다음으로 넘긴다
