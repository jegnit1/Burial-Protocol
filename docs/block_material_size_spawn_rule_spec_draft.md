# Burial Protocol - Block Material / Size / Spawn Rule Specification (Draft)

## 0. 목적

이 문서는 Burial Protocol의 블록 시스템을  
**재질(Material)**, **사이즈(Size)**, **출현 규칙(Spawn Rule)** 으로 분리하여 정의하기 위한 초안이다.

이 문서의 목적은 아래 항목들을 명확히 고정하는 것이다.

- 블록 베이스의 의미를 재질로 한정한다
- 블록 사이즈를 별도 축으로 분리한다
- 블록 체력, 보상, 위협도를 재질과 사이즈의 조합으로 계산한다
- 난이도와 스테이지에 따라 어떤 재질/사이즈가 얼마나 등장하는지 데이터로 제어한다
- 구글 스프레드시트에서 편집 가능한 구조를 기준으로 import / export 방향을 고정한다

이 문서는 구현 범위를 고정하기 위한 기준 문서이며,  
구현자는 임의로 블록 베이스에 사이즈 개념을 다시 섞지 않는다.

---

## 1. 핵심 원칙

### 1-1. 블록 베이스는 재질이다

블록 베이스는 블록의 **재질/성질**을 정의한다.

블록 베이스가 담당하는 것은 아래와 같다.

- 재질 ID
- HP 배율
- 보상 배율
- 기본 등장 가중치
- 색상/비주얼
- 특수 결과
- 특정 조건에서의 등장 제한

블록 베이스가 담당하지 않는 것은 아래와 같다.

- 가로 길이
- 세로 길이
- 면적
- 크기 기반 체력 증가량
- 크기 기반 회피 압박도

즉, 블록 베이스는  
"유리", "나무", "바위", "강철", "폭탄" 같은 **무엇으로 이루어졌는가**를 정의하는 축이다.

---

### 1-2. 블록 사이즈는 별도 축이다

블록 사이즈는 블록의 **가로/세로 크기**를 정의한다.

사이즈가 담당하는 것은 아래와 같다.

- 가로 길이
- 세로 길이
- 면적 기반 HP 배율
- 면적 기반 보상 배율
- 크기별 기본 등장 가중치
- 크기별 최소 등장 난이도
- 크기별 최소 등장 스테이지
- 크기별 상한/금지 조건

즉, 사이즈는  
"1x1", "2x1", "1x2", "2x2", "4x1" 같은 **얼마나 큰가**를 정의하는 축이다.

---

### 1-3. 실제 블록은 재질 × 사이즈의 조합이다

실제 런타임 블록은 아래 두 축의 조합으로 정의한다.

- material
- size

즉:

- 나무 1x1
- 나무 2x2
- 강철 1x2
- 폭탄 2x1

같은 식으로 구성된다.

구현상 블록은 단일 고정 타입이 아니라  
**재질 데이터와 사이즈 데이터의 조합으로 해석되는 인스턴스**여야 한다.

---

## 2. 재질(Material) 정의

## 2-1. 재질의 역할

재질은 블록의 **성질**을 정의한다.

예:

- 유리: 약하지만 파괴 시 주변 피해
- 황금: 보상이 높음
- 강철: 높은 HP 배율
- 폭탄: 모래화 대신 폭발

즉 재질은  
"이 블록을 어떻게 상대해야 하는가"에 대한 **질적 차이**를 만든다.

---

## 2-2. 재질이 가져야 하는 기본 필드

재질 데이터는 최소 아래 필드를 가진다.

- `material_id`
- `display_name`
- `hp_multiplier`
- `reward_multiplier`
- `base_spawn_weight`
- `special_result_type`
- `color_key`
- `is_enabled`

필요 시 아래 확장 필드를 둘 수 있다.

- `min_difficulty`
- `min_stage`
- `max_stage`
- `max_allowed_area`
- `max_allowed_width`
- `max_allowed_height`
- `notes`

---

## 2-3. 재질 배율의 의미

재질의 HP 배율은  
동일 사이즈 기준에서 어떤 재질이 더 단단한지를 정의한다.

예:

- 나무: 1.0
- 대리석: 1.5
- 시멘트: 2.0
- 강철: 3.0

즉 동일한 2x2 블록이라도  
강철은 나무보다 훨씬 더 오래 버틴다.

---

## 2-4. 재질 보상 배율의 의미

재질의 보상 배율은  
동일 사이즈 기준에서 어떤 재질이 더 높은 리턴을 주는지를 정의한다.

예:

- 나무: 1.0
- 황금: 2.0 ~ 5.0
- 강철: 1.3
- 폭탄: 특수 처리

보상 배율은 크기와 별개가 아니라  
크기 기반 기본 보상 위에 곱해지는 구조를 사용한다.

---

## 3. 사이즈(Size) 정의

## 3-1. 사이즈의 역할

사이즈는 블록의 **물리적 크기와 그에 따른 압박도**를 정의한다.

사이즈는 아래를 통해 게임플레이에 영향을 준다.

- 체력 요구량 증가
- 공간 점유량 증가
- 회피 공간 압박 증가
- 특정 난이도 이상에서만 등장
- 특정 스테이지 이상에서만 등장

---

## 3-2. 사이즈 기본 필드

사이즈 데이터는 최소 아래 필드를 가진다.

- `size_id`
- `width_u`
- `height_u`
- `area`
- `hp_multiplier`
- `reward_multiplier`
- `base_spawn_weight`
- `min_difficulty`
- `min_stage`
- `max_stage`
- `is_enabled`

필요 시 아래 확장 필드를 둘 수 있다.

- `tags`
- `notes`

---

## 3-3. 체력 배율은 1차적으로 면적 기준이다

블록 체력은 재질 배율보다 **사이즈 영향이 더 크다**.

기본 원칙은 아래와 같다.

- `1x1` 블록의 HP 배율은 `1`
- `2x1` 블록의 HP 배율은 `2`
- `1x2` 블록의 HP 배율은 `2`
- `2x2` 블록의 HP 배율은 `4`

즉, 사이즈 HP 배율은 기본적으로 아래 공식으로 계산한다.

`size_hp_multiplier = width_u * height_u`

이 규칙은 체력 계산의 1차 기준이다.

---

## 3-4. 가로 크기와 세로 크기의 플레이 감각은 다르다

체력 배율은 면적 기준으로 처리하되,  
플레이 감각에서 가로 크기와 세로 크기는 같은 의미가 아니다.

### 가로 증가의 의미

가로 크기가 커질수록 아래 영향을 준다.

- 플레이어의 피할 공간 감소
- 중앙 샤프트의 통과 가능 영역 축소
- 회피와 위치 조정 난이도 증가

즉, 가로 크기 증가는  
단순한 공격 스탯 요구치 증가가 아니라  
**회피 요구치**도 함께 증가시킨다.

### 세로 증가의 의미

세로 크기가 커질수록 아래 영향을 준다.

- 체력 총량 증가
- 처리 시간 증가
- 공격 스탯 요구치 증가

즉, 세로 크기 증가는  
회피 공간 압박보다는  
**공격 처리량 요구치**를 더 크게 만든다.

---

## 3-5. 현재 단계의 원칙

현재 단계에서는 아래 원칙을 사용한다.

- HP 계산은 면적 기준으로만 처리한다
- 가로/세로 차이는 스폰 제한과 플레이 감각으로 처리한다
- 가로가 큰 블록일수록 출현 조건을 더 보수적으로 잡는다

즉, `2x1`과 `1x2`는 같은 HP 배율을 가지지만,  
등장 확률과 최소 등장 구간은 다르게 조절될 수 있다.

---

## 4. 체력과 보상 계산 규칙

## 4-1. 최종 체력 계산

최종 블록 체력은 아래 순서로 계산한다.

1. 기본 단위 체력
2. 사이즈 HP 배율
3. 재질 HP 배율
4. 난이도 배율
5. 추가 스테이지 배율(필요 시)

권장 공식은 아래와 같다.

`final_hp = base_unit_hp × size_hp_multiplier × material_hp_multiplier × difficulty_hp_multiplier`

여기서:

- `base_unit_hp` 는 1U 기준 기본 체력이다
- `size_hp_multiplier = width_u * height_u`

---

## 4-2. 최종 보상 계산

최종 보상은 아래 순서로 계산한다.

1. 기본 단위 보상
2. 사이즈 보상 배율
3. 재질 보상 배율
4. 추가 보정(필요 시)

권장 공식은 아래와 같다.

`final_reward = base_unit_reward × size_reward_multiplier × material_reward_multiplier`

기본적으로 사이즈 보상 배율도  
면적 기준으로 처리하는 것을 권장한다.

즉:

- 1x1 = 1배
- 2x1 = 2배
- 2x2 = 4배

이 원칙을 기본으로 사용한다.

---

## 5. 출현 규칙(Spawn Rule)

## 5-1. 출현 확률은 재질과 사이즈 모두를 제어해야 한다

스테이지/난이도별 출현 확률은  
**사이즈만 결정하는 것이 아니라 특정 블록의 material/base/type도 함께 결정**해야 한다.

즉 현재 시점의 출현 규칙은 아래 두 축을 모두 다뤄야 한다.

- 어떤 사이즈가 등장 가능한가
- 어떤 재질이 등장 가능한가

따라서 출현 시스템은 단순히  
"현재 구간에서 큰 블록 확률만 올린다"로 끝나면 안 된다.

반드시 아래를 모두 제어해야 한다.

- size 가중치
- material 가중치
- 특정 size 금지
- 특정 material 금지
- 특정 material × size 조합 금지 또는 완화

---

## 5-2. 높은 난이도와 높은 스테이지일수록 큰 블록 확률 증가

기본 원칙은 아래와 같다.

- 난이도가 높아질수록 큰 사이즈 블록 출현 확률 증가
- 스테이지가 높아질수록 큰 사이즈 블록 출현 확률 증가

즉, 후반으로 갈수록  
큰 블록이 더 자주 등장해야 한다.

---

## 5-3. 높은 난이도와 높은 스테이지일수록 특정 재질도 더 적극적으로 등장 가능

기본 원칙은 아래와 같다.

- 난이도가 높아질수록 고위험 재질 등장 확률 증가
- 스테이지가 높아질수록 특수 재질 등장 확률 증가

예:

- 노말 초반: 나무, 바위 위주
- 노말 후반: 대리석, 시멘트 일부 등장
- 하드 이후: 강철, 황금, 폭탄 비중 증가

즉 재질의 출현 확률도  
스테이지/난이도에 따라 별도로 제어해야 한다.

---

## 5-4. 특정 사이즈의 최소 등장 조건이 있어야 한다

사이즈별로 아래 조건을 설정할 수 있어야 한다.

- 최소 난이도
- 최소 스테이지
- 최대 스테이지
- 비활성화 여부

예:

- `4x1` 은 노말 초반에 등장 금지
- `width 4U` 블록은 노말 낮은 스테이지부터 등장 불가
- `width + height = 8U` 급 블록은 하드부터 허용

즉 큰 블록은  
낮은 난이도와 초반 스테이지에서 제한되어야 한다.

---

## 5-5. 특정 재질의 최소 등장 조건도 있어야 한다

재질별로 아래 조건을 설정할 수 있어야 한다.

- 최소 난이도
- 최소 스테이지
- 특정 사이즈 이상 금지
- 특정 크기 구간에서만 등장
- 특정 구간에서 확률 증가

예:

- 폭탄은 노말 초반 금지
- 황금은 중반 이후부터 등장
- 강철은 하드 이후 비중 증가

즉 특정 재질도  
모든 구간에서 동일 확률로 등장하면 안 된다.

---

## 5-6. 재질과 사이즈의 조합 제한도 가능해야 한다

모든 재질이 모든 사이즈와 자유롭게 결합되면  
위협도와 밸런스가 급격히 무너질 수 있다.

따라서 아래 조합 제한이 가능해야 한다.

- 특정 재질은 일정 면적 이상 금지
- 특정 재질은 특정 가로 폭 이상 금지
- 특정 재질은 특정 세로 높이 이상 금지
- 특정 난이도 이하에서 특정 조합 금지

예:

- 폭탄은 초반에 1x1, 2x1까지만 허용
- 황금은 지나치게 큰 사이즈 확률을 낮춤
- 유리는 대형 사이즈 출현을 제한 가능

---

## 5-7. 출현 결정 순서

권장 출현 순서는 아래와 같다.

1. 현재 난이도 / 스테이지 확인
2. 현재 구간에서 허용된 사이즈 후보 수집
3. 현재 구간에서 허용된 재질 후보 수집
4. 사이즈 가중치로 사이즈 결정
5. 재질 가중치로 재질 결정
6. 재질 × 사이즈 조합 유효성 검증
7. 유효하면 블록 생성
8. 유효하지 않으면 재선정

이 방식을 기본으로 한다.

---

## 6. 스프레드시트 기반 데이터 관리

## 6-1. 스프레드시트 관리 원칙

구글 스프레드시트는  
블록 재질/사이즈/출현 규칙을 편집하는 **주 편집 도구**로 사용한다.

목표는 아래와 같다.

- 기획자가 시트에서 밸런스를 조정할 수 있다
- 게임은 시트 데이터를 import 해서 사용한다
- 추후 필요하면 export 도 지원한다

---

## 6-2. 권장 시트 구성

최소 아래 시트 구성을 사용한다.

### A. `block_materials`

재질 정의 시트

필드 예:

- material_id
- display_name
- hp_multiplier
- reward_multiplier
- base_spawn_weight
- special_result_type
- color_key
- min_difficulty
- min_stage
- max_stage
- max_allowed_area
- max_allowed_width
- max_allowed_height
- is_enabled
- notes

---

### B. `block_sizes`

사이즈 정의 시트

필드 예:

- size_id
- width_u
- height_u
- area
- hp_multiplier
- reward_multiplier
- base_spawn_weight
- min_difficulty
- min_stage
- max_stage
- is_enabled
- tags
- notes

---

### C. `block_spawn_rules`

난이도/스테이지별 출현 규칙 시트

필드 예:

- difficulty_id
- stage_from
- stage_to
- material_id
- size_id
- spawn_weight
- enabled
- notes

이 시트는 아래 역할을 담당한다.

- 특정 구간의 등장 확률 조정
- 특정 material 출현 제어
- 특정 size 출현 제어
- 특정 material × size 조합 제어

---

## 6-3. 시트 작성 기준

시트는 아래 기준을 따른다.

- 하나의 행은 하나의 데이터 단위를 표현한다
- 수식 자동 생성이 가능한 값은 중복 입력을 최소화한다
- 사람이 읽기 쉬운 이름과 내부 ID를 함께 둔다
- 주석/비고 컬럼을 별도 둔다

---

## 7. Import / Export 지원 방향

## 7-1. import 우선

현재 단계에서는 **import를 우선 지원**한다.

즉:

- 시트 작성
- 데이터 읽기
- 내부 JSON / 리소스 변환
- 런타임 로딩

순서로 진행한다.

---

## 7-2. export는 후순위

export는 추후 필요 시 지원한다.

현재 단계에서는 아래 방향이면 충분하다.

- 시트 → 게임 데이터 반영 가능
- 게임 데이터 validation 가능
- 잘못된 값 경고 가능

즉 초기 목표는  
게임 내 수정 내용을 다시 시트로 보내는 것이 아니라  
시트에서 편집한 내용을 안전하게 게임에 반영하는 것이다.

---

## 7-3. import 대상 데이터

import 대상은 최소 아래 3종이다.

- materials
- sizes
- spawn_rules

이 데이터는 import 후 내부적으로 아래 형태로 변환될 수 있다.

- json
- csv 파싱 결과
- Godot Resource
- autoload data cache

단, 최종 런타임은  
시트 자체를 직접 읽는 구조보다  
한 번 정제된 내부 데이터 구조를 사용하는 것을 권장한다.

---

## 8. 코드 구조 권장안

## 8-1. 권장 데이터 구조

구현은 최소 아래 개념을 가진다.

- `BlockMaterialData`
- `BlockSizeData`
- `BlockSpawnRuleData`
- `BlockResolvedDefinition`
- `BlockDataImporter`
- `BlockSpawnResolver`

---

## 8-2. BlockResolvedDefinition의 역할

실제 런타임에서 생성되는 블록은  
최소 아래 정보를 가진다.

- material_id
- size_id
- width_u
- height_u
- final_hp
- final_reward
- special_result_type
- color_key

즉, 생성 직전에  
재질/사이즈/출현 규칙이 합쳐진 최종 결과를 만든다.

---

## 8-3. 기존 코드 수정 방향

프로젝트 코드는 아래 방향으로 수정한다.

- 블록 베이스 데이터에서 사이즈 제거
- 사이즈는 별도 데이터에서 참조
- HP 계산은 면적 기반으로 재구성
- 스폰 로직은 material/size를 분리 선정
- 난이도/스테이지 조건으로 spawn rule을 해석
- import된 시트 데이터를 기준으로 블록 생성

---

## 9. 현재 단계에서 반드시 지킬 것

아래는 반드시 지킨다.

- 블록 베이스에 사이즈 개념을 다시 넣지 않는다
- 재질과 사이즈를 하나의 고정 블록 타입으로 합치지 않는다
- 난이도/스테이지 제어를 코드 하드코딩으로 늘리지 않는다
- 스프레드시트 구조와 코드 구조가 같은 개념 체계를 공유해야 한다

---

## 10. 현재 단계에서 보류 가능한 것

아래는 추후 확장 가능하다.

- 재질과 속성의 추가 분리
- 보스 전용 블록군
- 특정 Day 전용 스폰 규칙
- 계절/이벤트형 블록 테이블
- 스프레드시트 export 자동화

---

## 12. 결론

Burial Protocol의 블록 시스템은 앞으로 아래 구조를 따른다.

- 블록 베이스 = 재질
- 블록 사이즈 = 별도 축
- 실제 블록 = 재질 × 사이즈
- 블록 체력은 사이즈 영향이 더 크고, 재질은 그 위에 배율을 준다
- 난이도/스테이지별 출현 규칙은 사이즈뿐 아니라 재질도 함께 결정한다
- 재질/사이즈/출현 규칙은 구글 스프레드시트에서 편집 가능해야 한다
- 게임은 해당 시트 데이터를 import 하여 런타임에 사용한다

즉, 이 시스템의 핵심은  
블록을 하나의 고정 타입으로 보지 않고,  
**재질 / 크기 / 출현 규칙을 분리한 데이터 조합형 위협 오브젝트**로 취급하는 데 있다.
