# Burial Protocol - Scaling Change Note

## 0. 목적

이 문서는 스케일 변경 작업을 할 때
무엇을 먼저 계산하고 무엇을 함께 조정해야 하는지 정리하는 보조 메모다.
권위 문서는 아니지만,
`UNIT_SIZE`, `VIEWPORT_SIZE`, 월드 크기, HUD, px 기반 이동 수치를 같이 다뤄야 한다는 점을 분명히 하기 위해 유지한다.

---

## 1. 현재 적용 스냅샷

현재 구현 기준 스케일은 아래와 같다.

| 항목 | 현재 값 |
|---|---:|
| 뷰포트 | 1920 x 1080 |
| UNIT_SIZE / CELL_SIZE | 64 |
| PLAYER_SIZE | 128 x 128 |
| CENTER_COLUMNS | 10 |
| WALL_COLUMNS | 10 |
| WORLD_COLUMNS | 30 |
| WORLD_ROWS | 200 |
| WORLD_PIXEL_WIDTH | 1920 |
| WORLD_PIXEL_HEIGHT | 12800 |

즉, 현재는
가로 30U 전체를 한 화면에 고정하고,
세로는 매우 큰 월드를 두고 카메라가 Y축만 따라가는 방향을 사용하고 있다.

---

## 2. 이번 작업에서 배운 핵심 원칙

### 2-1. 체감 크기 요구는 카메라 수식보다 월드/플레이어 수치부터 본다

사용자가
"1U를 더 크게 보이게 하고 싶다"
"채굴 깊이를 늘리고 싶다"
처럼 명확한 체감 목표를 말하면,
먼저 아래를 직접 계산한다.

- `UNIT_SIZE`
- `WORLD_COLUMNS`
- `WORLD_ROWS`
- `WORLD_PIXEL_WIDTH`
- `WORLD_PIXEL_HEIGHT`
- `VIEWPORT_SIZE`
- `PLAYER_SIZE`

### 2-2. 아래 값은 한 묶음이다

스케일 변경 시 아래는 같이 봐야 한다.

- `UNIT_SIZE`
- `CELL_SIZE`
- 뷰포트
- HUD 높이와 마진
- 폰트 크기
- 플레이어 이동 속도
- 점프 속도
- 중력
- 블록 낙하 속도
- 모래 셀 크기
- 공격/채굴 판정 기준 바디 크기

### 2-3. px 기반 이동값은 비율로 같이 스케일링한다

예를 들어
`48 -> 64`로 바뀌면
`64 / 48 = 1.333...` 비율로
이동/점프/낙하 속도 계열도 같이 보는 편이 현재 체감 보존에 유리하다.

---

## 3. 추천 절차

스케일을 바꿀 때는 아래 순서를 권장한다.

1. 현재 `GameConstants.gd` 수치를 읽는다
2. 사용자 목표를 숫자로 바꾼다
3. 새 월드 픽셀 크기를 계산한다
4. `project.godot` 뷰포트를 맞춘다
5. `GameConstants.gd` 상수를 함께 조정한다
6. px 기반 이동/낙하 수치를 비율로 맞춘다
7. 플레이어 충돌 바디와 공격/채굴 범위의 기준점을 함께 점검한다
8. HUD와 상태 텍스트 가독성을 확인한다
9. 마지막에 카메라 추적 방식과 스폰 기준을 점검한다

핵심은
"카메라 fit으로 체감 크기를 억지로 만들기보다,
월드, 플레이어, 뷰포트를 먼저 맞춘다"
는 점이다.

---

## 4. 현재 구현이 시사하는 것

현재 구현은 아래 철학에 가깝다.

- 가로 30U는 항상 전부 보여준다
- 플레이어는 2U 크기로 키우고, 바디는 `COLLISION_INSET`을 반영해 따로 잡는다
- 공격/채굴 프리뷰와 실제 범위도 이 충돌 바디 기준에 맞춰야 한다
- 카메라는 X를 고정하고 Y만 따라간다
- 낙하 블록은 현재 보이는 상단 시야 밖에서 스폰된다

즉, 스케일 변경 작업은
카메라 트릭보다
월드 구조와 수치의 일괄 조정 문제로 보는 편이 정확하다.

---

## 5. 현재 남은 주의점

- 현재 세로 월드는 충분히 큰 고정 높이 월드이지만, 완전한 무한 확장 구조는 아직 아니다
- 채굴/공격 범위는 이제 충돌 바디 기준 정렬을 사용하지만, 벽 채굴 순서 문제는 별도 후속 수정이 필요하다
- 화면 스펙 문서를 건드릴 때는 `CODEX_SCREEN_LAYOUT_SPEC.md`를 최상위 기준으로 다시 대조해야 한다

---

## 6. 개인 메모

- 스케일 변경 요청이 오면 카메라 수식부터 만지지 말 것
- 먼저 `GameConstants`, `WorldGrid`, `Main`, `Player`를 같이 읽을 것
- 체감 목표를 숫자로 떨어뜨린 뒤 문서와 코드에 같이 반영할 것
- 변경 후에는 문서도 같은 턴에 갱신할 것
