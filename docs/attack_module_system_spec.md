# Burial Protocol - Attack Module System Specification

## 0. 목적

이 문서는 Burial Protocol의 공격 모듈 시스템을 현재 최신 구현 기준 위에 추가하기 위한 공식 작업 문서다.

현재 프로젝트는 아래 상태를 가진다.

- 플레이어 공격/채굴 입력 분리 완료
- 공격 판정은 플레이어 기준 shape data 계산 구조로 정리됨
- Day 진행 및 인터미션 상점 진입 구조 존재
- 런타임 공격/채굴/이동/체력 보너스 구조 존재
- 단일 프로필 저장 구조 존재
- 상점 UI는 현재 placeholder 상태이며 실제 상품 구조는 아직 미구현

따라서 이번 문서의 목적은 아래를 고정하는 것이다.

- 공격 모듈의 개념 정의
- 현재 확정 공격 모듈 목록과 수치 확정
- 런 내 구매 / 장착 / 교체 규칙 정의
- 플레이어 주변 부유 비주얼 규칙 정의
- 상인 UI 및 추후 업그레이드 시스템 확장 여지 정의
- 이번 구현 범위와 비범위 명확화

이 문서는
공격 모듈 시스템을 "런 내 무기 장비 시스템"으로 확정하기 위한 기준 문서이며,
구현자는 임의로 범위를 확장하지 않는다.

---

## 1. 현재 구현 기준선

현재 최신 코드 기준선은 아래와 같다.

- `Player.gd` 는 공격 입력과 채굴 입력을 분리해 각각 버퍼/쿨다운/방향 처리를 수행한다.
- `Player.gd` 는 공격 shape 와 채굴 shape 를 별도 계산한다.
- `Main.gd` 는 실제 공격 판정과 Day 진행, 인터미션 상점 진입 흐름을 관리한다.
- `DayShopUI.gd` 는 현재 placeholder 상태지만 인터미션 상점 진입점은 이미 연결되어 있다.
- `GameState.gd` 는 런타임 공격력/공격속도/공격범위 등의 보너스 수치를 관리한다.
- `GameState.gd` 는 단일 프로필 저장 구조를 가진다.
- `GameData.gd` 는 블록/스테이지 데이터처럼 리소스 기반 데이터를 읽는 구조다.

즉, 공격 모듈 시스템은
기존 전투 구조를 대체하는 것이 아니라,
현재 공격 처리 위에 "런 내 장비 데이터 레이어"를 추가하는 방식으로 구현한다.

---

## 2. 공격 모듈의 개념

공격 모듈은 Burial Protocol에서 사실상의 무기 장비에 해당한다.

플레이어 본체는 아래 역할을 담당한다.

- 이동
- 점프
- 생존
- 채굴
- 위치 선정

실제 공격 수행은
플레이어 주변에 부유하는 공격 모듈이 담당한다.

따라서 공격 모듈은 아래 성질을 가진다.

- 기본 공격 판정의 범위를 결정한다
- 공격 타입을 가진다
- 공격 데미지 배율을 가진다
- 공격속도 배율을 가진다
- 장착 시 전용 소형 에셋이 플레이어 주변에 부유 표시된다

중요한 점은 아래와 같다.

- 공격 모듈은 "스킬"이 아니라 "런 내 장착 장비"다
- 현재는 공격 모듈 슬롯 1개만 사용한다
- 공격 모듈은 플레이어의 기본 공격을 완전히 새 로직으로 바꾸지 않는다
- 기존 공격 입력과 판정 구조를 유지한 채, 범위/배율/비주얼만 장비 기준으로 덮어쓴다

---

## 3. 용어 정의

### 3.1 공격 모듈

플레이어의 현재 기본 공격 형태를 결정하는 런 내 장비 슬롯 아이템.

### 3.2 보유 공격 모듈

현재 런 안에서 이미 구매 또는 획득되어 장착 가능한 공격 모듈 목록.

### 3.3 장착 공격 모듈

현재 플레이어가 실제로 사용 중인 공격 모듈.

### 3.4 공격 모듈 비주얼

장착된 공격 모듈을 플레이어 주변에 표시하는 부유 오브젝트.

### 3.5 공격 모듈 업그레이드

같은 런 안에서 현재 보유/장착 중인 공격 모듈의 성능을 추가 강화하는 확장 시스템.
이번 문서에서는 실제 구현 대신 확장 여지만 남긴다.

---

## 4. 이번 Phase에서 확정하는 공격 모듈 목록

아래 5종을 1차 공격 모듈 목록으로 확정한다.

| NAME | 범위 | 타입 | 데미지 배율 | 공격속도 배율 |
|---|---|---|---:|---:|
| 소드 모듈 | 1U x 1U | 근거리 | 1.0 | 1.0 |
| 단검 모듈 | 0.5U x 1U | 근거리 | 0.8 | 1.4 |
| 랜스 모듈 | 2.5U x 0.5U | 근거리 | 1.0 | 0.85 |
| 도끼 모듈 | 1U x 2U | 근거리 | 1.2 | 0.8 |
| 대검 모듈 | 1.5U x 1U | 근거리 | 1.5 | 0.65 |

### 4.1 역할 의도

- 소드 모듈: 기준형
- 단검 모듈: 짧은 범위, 빠른 연타형
- 랜스 모듈: 긴 리치, 얇은 전방 커버형
- 도끼 모듈: 세로 커버형
- 대검 모듈: 강한 한 방형

### 4.2 기본 시작 모듈

현재 기준 기본 시작 공격 모듈은 `소드 모듈`로 한다.

이유는 아래와 같다.

- 현재 프로젝트의 기본 공격 판정 감각이 1U x 1U 기준에 가장 가깝다
- 기존 플레이 감각을 가장 적게 흔든다
- 다른 모듈의 차이를 비교하기 위한 기준 역할을 맡기 쉽다

## 5. 장착 규칙

### 5.1 장착 슬롯 수

현재는 공격 모듈 슬롯을 1개만 둔다.

- 한 번에 공격 모듈 1개만 장착 가능
- 복수 공격 모듈 동시 장착은 이번 범위에서 제외

### 5.2 기본 시작 상태

런 시작 시 플레이어는 아래 상태로 시작한다.

- 기본 장착 공격 모듈: `소드 모듈`
- 기본 보유 공격 모듈 목록: `소드 모듈` 1개만 보유

### 5.3 구매와 교체

현재 기준 공격 모듈은 **런 내 상인에게 구매**하는 장비다.

즉,

- 허브에서 미리 보유/장착하지 않는다
- 인터미션 단계의 상인 UI에서 구매 및 교체한다
- 현재 런 안에서만 보유 상태를 유지한다
- 런 종료 시 공격 모듈 보유/장착 상태는 초기화된다

### 5.4 장착 변경 위치

현재 기준 장착 변경은 런 내 상인에게서만 가능하다.

즉,

- 런 도중 임의 교체 없음
- 허브 교체 시스템은 이번 범위에서 제외
- 상인 UI가 실제 장착 변경 진입점이다

### 5.5 장착 변경 비용

이번 기준에서는 아래를 따른다.

- 신규 공격 모듈 구매에는 비용이 든다
- 이미 구매한 공격 모듈로의 교체는 무료로 둔다

이유는 아래와 같다.

- 런 내 전략 전환은 허용하되
- 같은 런 안에서 이미 확보한 장비를 다시 쓰는 데 불필요한 비용을 두지 않는다

## 6. 계산 원칙

공격 모듈은 기존 플레이어 기본 공격 위에 배율과 판정 형태를 덧씌운다.

### 6.1 최종 공격력

최종 공격력 계산 기준은 아래를 따른다.

`final_attack_damage = floor(current_attack_damage * module_damage_multiplier)`

여기서 `current_attack_damage` 는 아래를 포함한 값이다.

- 기본 공격력
- 런타임 공격력 증가
- 기타 현재 적용 중인 공격력 보너스

### 6.2 최종 공격속도

공격속도 배율은 현재 쿨다운 기반 구조를 유지한 채 아래처럼 적용한다.

`final_attack_cooldown = current_attack_cooldown / module_attack_speed_multiplier`

즉,

- 공격속도 배율이 높을수록 더 빠르다
- 공격속도 배율이 낮을수록 더 느리다

### 6.3 최종 공격 범위

현재 공격 shape 계산 구조를 유지하되,
장착 모듈의 범위 수치를 사용한다.

즉,

- 기본 `1U x 1U` 고정이 아니다
- 장착된 공격 모듈의 `range_width_u`, `range_height_u` 가 실제 shape 크기를 결정한다

### 6.4 공격 방향

공격 방향 해석은 현재 플레이어 기준 구조를 그대로 유지한다.

즉,

- 마우스 방향 기반 공격 방향 해석 유지
- 모듈 장착이 공격 방향 해석 방식을 바꾸지 않는다

---

## 7. 데이터 구조 원칙

공격 모듈은 하드코딩 분기보다 데이터 기반 구조를 우선한다.

### 7.1 권장 데이터 필드

공격 모듈 데이터는 최소 아래 필드를 가진다.

- `module_id`
- `display_name`
- `module_type`
- `range_width_u`
- `range_height_u`
- `damage_multiplier`
- `attack_speed_multiplier`
- `visual_scene_path`
- `shop_price_gold`
- `default_start_module`
- `description`

### 7.2 예시

- `module_id = "sword_module"`
- `display_name = "소드 모듈"`
- `module_type = "melee"`
- `range_width_u = 1.0`
- `range_height_u = 1.0`
- `damage_multiplier = 1.0`
- `attack_speed_multiplier = 1.0`
- `visual_scene_path = "res://..."`
- `shop_price_gold = 0`
- `default_start_module = true`

### 7.3 데이터 관리 방향

블록/스테이지 데이터와 마찬가지로,
공격 모듈도 추후 `GameData` 계층에서 읽을 수 있는 리소스 또는 데이터 테이블 구조를 권장한다.

즉,

- `GameConstants` 에 모든 모듈 내용을 직접 하드코딩하지 않는다
- 공용 시스템은 데이터 접근 함수만 가진다
- 실제 모듈 목록은 데이터 리소스가 소유하는 방향을 우선한다

---

## 8. 저장 구조 원칙

공격 모듈 시스템은 현재 **런 내 상태**로 관리한다.

### 8.1 런타임 필드

최소 아래 필드를 런타임 상태로 둔다.

- `owned_attack_module_ids`
- `equipped_attack_module_id`

### 8.2 기본값

런 시작 시 기본값은 아래를 따른다.

- 보유 목록: `["sword_module"]`
- 장착 ID: `"sword_module"`

### 8.3 저장 범위

현재 기준에서 아래를 따른다.

- 런 종료 후 공격 모듈 보유 목록은 저장하지 않는다
- 런 종료 후 장착 상태도 저장하지 않는다
- 즉, 공격 모듈은 영구 메타 자산이 아니다

주의:
추후 메타 성장과 연동해 "런 시작 시 해금 가능한 스타팅 모듈 확장"이 필요해질 수 있으나,
그 개념은 이번 문서 범위에 포함하지 않는다.

---

## 9. 플레이어 주변 부유 비주얼 규칙

장착된 공격 모듈은 플레이어 주변에 떠 있는 시각 오브젝트로 표시한다.

### 9.1 목적

- 현재 장착 무기를 즉시 식별 가능하게 한다
- 플레이어 본체 직접 공격 모션이 약한 구조를 보완한다
- 상인 구매/교체 결과를 시각적으로 분명히 보여준다

### 9.2 기본 규칙

- 장착 모듈 비주얼은 현재 1개만 표시한다
- 플레이어 중심 주위를 공전하거나 둥둥 떠다닌다
- 플레이어와 완전히 겹치지 않는다
- 공격 시에는 현재 공격 방향 쪽으로 짧게 반응한다

### 9.3 에셋 기준

현재 기준 기본 원칙은 아래와 같다.

- 부유 비주얼은 단순 아이콘이 아니라 **별도 소형 에셋**을 사용한다
- UI 아이콘과 월드 비주얼은 동일 파일일 필요가 없다
- 상점 목록용 아이콘과 플레이어 주변 부유용 에셋은 분리 가능하게 설계한다

즉, 이번 단계에서는
"상점 아이콘"과 "월드 부유 에셋"을 분리할 수 있는 구조를 연다.

### 9.4 권장 기본 연출

기본 상태:
- 플레이어 중심 기준 반지름 28px ~ 48px 부근 유지
- 느린 공전 또는 타원 궤도 이동
- 소폭 상하 bobbing

공격 시:
- 공격 방향으로 짧게 치고 나감
- 또는 해당 방향으로 빠르게 보간
- 공격 종료 후 원래 공전 위치로 복귀

### 9.5 이번 최소 구현 범위

이번 범위에서는 아래까지만 구현한다.

- 장착 모듈 비주얼 1개 부유 표시
- idle 공전 또는 둥둥 떠다니기
- 공격 시 짧은 방향 반응
- 모듈 변경 시 비주얼 교체

이번 범위에서 아래는 하지 않는다.

- 복수 모듈 공전
- 잔상/파티클 강화 연출
- 모듈별 완전 다른 공격 애니메이션
- 독립 AI형 위성 동작

---

## 10. 상인 연동 기준

현재 `DayShopUI` 는 placeholder 상태다.
공격 모듈 시스템은 추후 인터미션 상인과 연결한다.

### 10.1 현재 기준 상인 기능 목표

- 아직 구매하지 않은 공격 모듈 구매 가능
- 이미 구매한 공격 모듈 장착 가능
- 현재 장착 공격 모듈 표시 가능
- 현재 런 안에서 보유한 공격 모듈 목록 표시 가능

### 10.2 구매 규칙

현재 최소 기준은 아래를 따른다.

- 구매 비용은 골드 사용
- 구매 후 현재 런 안에서만 보유
- 장착 변경 비용 없음
- 인터미션 상인에게서만 구매 및 변경 가능

### 10.3 이번 범위와 다음 범위

이번 범위:
- 데이터 구조 준비
- 장착 상태 반영
- 부유 비주얼 반영
- 상인 UI가 이 데이터를 사용할 수 있게 훅 준비

다음 범위:
- 실제 모듈 목록 표시
- 가격 및 구매 처리
- 장착/교체 버튼
- 구매 후 즉시 장착 여부 UX
- 모듈 설명 툴팁

---

## 11. 추후 업그레이드/강화 확장 여지

공격 모듈은 추후 같은 런 안에서 업그레이드 또는 강화될 수 있다.

### 11.1 현재 문서에서의 원칙

- 업그레이드/강화 시스템의 실제 구현은 이번 범위에서 제외한다
- 다만 공격 모듈 데이터와 런타임 상태 구조는 추후 강화 단계를 붙일 수 있게 열어 둔다

### 11.2 확장 가능한 방향

예시:

- 데미지 강화
- 공격속도 강화
- 범위 강화
- 특수 판정 추가
- 모듈 전용 파생 강화

### 11.3 구조 준비 원칙

추후 확장을 위해 아래 같은 필드를 붙일 수 있게 설계한다.

- `module_level`
- `upgrade_stage`
- `runtime_bonus_damage_multiplier`
- `runtime_bonus_attack_speed_multiplier`
- `runtime_bonus_range_multiplier`

중요:
이번 단계에서는 이 필드를 실제 게임 규칙에 연결하지 않아도 된다.
단, 구조적으로 붙일 수 없는 방향으로 구현하지 않는다.

## 12. 코드 책임 분리 권장안

### 12.1 `GameState`

- 현재 런 안에서의 보유 공격 모듈 목록 관리
- 현재 장착 공격 모듈 ID 관리
- 런 시작 시 기본값 초기화
- 추후 업그레이드 런타임 상태 확장 가능 구조 유지

### 12.2 `GameData`

- 공격 모듈 데이터 리소스 로딩
- ID 기반 정의 반환
- 추후 상인 표시용 목록 제공

### 12.3 `Player`

- 장착 모듈 비주얼 노드 관리
- 공격 direction 기준 비주얼 반응 처리
- 공격 shape 계산 시 장착 모듈 범위 반영

### 12.4 `Main`

- 실제 공격 처리 시 장착 모듈 데미지/공속 배율 반영
- 상인 구매/장착 결과를 실제 런에 반영
- DayShopUI 와 GameState 사이 연결

### 12.5 `DayShopUI`

- 공격 모듈 구매 목록 표시
- 보유 여부 표시
- 현재 장착 여부 표시
- 구매 / 장착 / 다음 Day 이동 진입점 제공

## 13. 이번 작업 범위

이번 작업에서 반드시 구현할 것:

- 공격 모듈 데이터 구조 도입
- 기본 시작 공격 모듈 개념 도입
- 런 내 장착 공격 모듈 ID 반영 구조 도입
- 런 내 보유 공격 모듈 목록 구조 도입
- 공격 shape 계산에 모듈 범위 반영
- 공격 데미지/공격속도에 모듈 배율 반영
- 플레이어 주변 부유 비주얼 1개 구현

이번 작업에서 하지 않을 것:

- 기능 모듈 본 구현
- 강화 모듈 본 구현
- 복수 공격 모듈 슬롯
- 자동 공격형 모듈
- 투사체형 모듈
- 허브 장착 화면
- 영구 해금형 공격 모듈 메타 구조
- 실제 모듈 업그레이드/강화 규칙 구현
- 고급 상인 UI 연출

## 14. 테스트 기준

### 14.1 기본 시작 상태

- 런 시작 시 기본 공격 모듈이 장착되어 있어야 한다
- 기본 장착 공격 모듈은 소드 모듈이어야 한다
- 런 시작 시 기본 보유 목록에는 소드 모듈만 있어야 한다

### 14.2 공격 범위 반영

- 소드/단검/랜스/도끼/대검 장착 시 공격 shape 크기가 각각 달라져야 한다
- 범위 차이가 시각 프리뷰와 실제 판정 모두에 반영되어야 한다

### 14.3 공격 성능 반영

- 장착 모듈에 따라 실제 공격 데미지가 달라져야 한다
- 장착 모듈에 따라 실제 공격 쿨다운이 달라져야 한다

### 14.4 비주얼 반영

- 장착 모듈 전용 소형 에셋이 플레이어 주변에 떠 있어야 한다
- 공격 시 비주얼이 현재 공격 방향 쪽으로 짧게 반응해야 한다
- 모듈 교체 시 부유 비주얼도 즉시 교체되어야 한다

### 14.5 상인 연동 준비

- 현재 런 안에서 보유한 공격 모듈 목록을 읽고 표시할 수 있어야 한다
- 현재 장착 모듈 ID를 읽고 표시할 수 있어야 한다
- 구매 후 현재 런 안에서만 보유 상태가 유지되어야 한다

### 14.6 강화 확장 준비

- 현재 구조가 추후 모듈 강화 단계를 붙여도 완전히 갈아엎지 않도록 유지되어야 한다

## 16. 금지 사항

아래 작업은 이번 범위에서 금지한다.

- 공격 모듈 때문에 기존 공격/채굴 입력 구조를 다시 합치는 것
- 장비 데이터 없이 모듈별 하드코딩 분기를 대량 추가하는 것
- 복수 공격 모듈 슬롯을 이번 단계에서 먼저 도입하는 것
- 영구 해금형 메타 장비 시스템으로 먼저 확장하는 것
- 상인 구조가 확정되기 전에 허브/런타임/상인 장착 규칙을 모두 동시에 확장하는 것
- 공격 모듈을 별도 자동 AI 전투 시스템처럼 먼저 구현하는 것
- 업그레이드 확장 여지만 있으면 되는 단계에서 강화 규칙까지 함께 확정하는 것

## 17. 결론

공격 모듈 시스템은 Burial Protocol에서 사실상의 런 내 무기 장비 시스템이다.

이번 단계의 목표는 아래와 같다.

- 기존 공격 입력과 판정 구조를 유지한 채
- 현재 기본 공격을 데이터 기반 장비 시스템으로 확장하고
- 장착된 모듈을 플레이어 주변 부유 비주얼로 표현하며
- 런 내 상인 구매/교체와 자연스럽게 연결 가능한 구조를 만들고
- 추후 업그레이드/강화 시스템을 붙일 여지를 남긴다

즉, 이번 작업은
단순히 공격 범위를 바꾸는 기능 추가가 아니라,
이후 기능 모듈 / 강화 모듈 / 상인 메타 시스템으로 확장될 장비 시스템의 첫 고정 단계다.
