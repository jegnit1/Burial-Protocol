# Burial Protocol - Gameplay Systems Specification

## 0. 목적

이 문서는 Burial Protocol의 현재 플레이 중 시스템을 한 문서에 정리한다.
플레이어 이동, 점프, 대시, 공격, 채굴, 피격, 블록 낙하, 파괴/분해, 모래 생성/흐름/채굴/밀기 규칙을
현재 코드 기준으로 명확히 고정하는 것이 목적이다.

이 문서는 다음을 다룬다.

- 입력 체계
- 플레이어 이동 / 점프 / 벽 기동 / 빠른 낙하 / 대시
- 공격과 채굴 판정
- 낙하 블록의 낙하 / 파괴 / 분해 / 압착 규칙
- 모래 생성 / 흐름 / 채굴 / 밀기 / 점프 공간 확보
- HUD와 상태 문구에 연결되는 런타임 정보

---

## 1. 한 줄 정의

현재 Burial Protocol의 플레이 시스템은

**좌클릭 360도 공격과 우클릭 방향 채굴, 점프/벽 점프/더블 점프/3방향 대시, 낙하 블록 파괴와 모래 누적, 그리고 모래 지형 활용과 채굴을 한 루프 안에 묶은 생존형 액션 구조**

로 요약할 수 있다.

---

## 2. 현재 입력 체계

현재 기본 입력은 아래와 같다.

| 행동 | 입력 |
|---|---|
| 좌이동 | `A`, `←` |
| 우이동 | `D`, `→` |
| 점프 | `W`, `↑`, `Space` |
| 빠른 낙하 | `S`, `↓` |
| 공격 | 마우스 좌클릭 |
| 채굴 | 마우스 우클릭 |
| 재시작/런 종료 | `R` |
| 디버그 토글 | `Tab` |

핵심은 아래와 같다.

- 공격과 채굴은 별도 입력이다
- 공격과 채굴은 별도 쿨다운과 입력 버퍼를 가진다
- 더 이상 기본 액션 하나에 공격/채굴이 묶이지 않는다

---

## 3. 플레이어 이동 및 기동

### 3-1. 기본 이동

현재 이동 감각은 관성보다 즉답성에 가깝다.

| 항목 | 값 |
|---|---:|
| 지상 이동 속도 | `426 px/s` |
| 공중 이동 속도 | `373 px/s` |
| 모래 위 이동 배율 | `0.62` |

즉, 현재는 즉시 반응하는 좌우 이동이 기본이다.

### 3-2. 점프

| 항목 | 값 |
|---|---:|
| 점프 속도 | `-853` |
| 추가 점프 수 | `1` |
| 점프 버퍼 | `0.14초` |
| 코요테 타임 | `0.1초` |

현재는 아래가 가능하다.

- 일반 점프
- 더블 점프
- 벽 점프

### 3-3. 벽 기동

| 항목 | 값 |
|---|---:|
| 벽 점프 가로 속도 | `480` |
| 벽 슬라이드 최대 하강 속도 | `200` |

벽 접촉 중 점프하면 벽 점프가 가능하고,
하강 중이면 벽 슬라이드 속도 제한이 걸린다.

### 3-4. 빠른 낙하

| 항목 | 값 |
|---|---:|
| 빠른 낙하 추가 가속 | `2933` |
| 빠른 낙하 최대 속도 | `1306` |

공중에서 아래 입력 시 낙하를 더 빠르게 만든다.

---

## 4. 대시 시스템

현재 플레이어는 좌 / 우 / 하 방향 대시를 가진다.

### 4-1. 입력 방식

- 같은 방향키를 짧은 시간 안에 두 번 누르면 대시 요청 발생
- 좌 / 우 / 하만 지원
- 위 대시는 비활성

### 4-2. 현재 수치

| 항목 | 값 |
|---|---:|
| 대시 거리 | `4U` |
| 더블탭 허용 시간 | `0.22초` |
| 대시 지속시간 | `0.08초` |
| 대시 쿨다운 | `0.45초` |

### 4-3. 의미

- 좌 / 우 대시: 회피, 위치 재조정, 모래 더미 옆 탈출
- 하 대시: 빠른 하강, 상부 압박 회피, 착지 위치 조정

현재 대시는 독립된 회피 수단이며,
일반 이동 / 점프 / 더블점프와 분리된 감각을 가진다.

---

## 5. 공격 시스템

### 5-1. 기본 원칙

현재 공격은 아래 성질을 가진다.

- 좌클릭 입력
- 히트스캔
- 다중 타격 가능
- 360도 방향 사용
- 플레이어 기준 판정
- 낙하 블록 전용 타격

### 5-2. 현재 수치

| 항목 | 값 |
|---|---:|
| 공격 피해 | `10` |
| 공격 범위 가로 길이 | `1U` |
| 공격 범위 세로 두께 | `1U` |
| 공격 쿨다운 | `0.1초` |
| 공격 입력 버퍼 | `0.12초` |

### 5-3. 공격 방향

현재 공격 방향은
플레이어 몸체 중심에서 마우스 위치를 향하는 벡터를 그대로 사용한다.

즉:

- 예전 4방향 양자화 구조가 아니다
- 대각선 포함 전 방향 공격이 가능하다
- 입력 길이가 너무 짧으면 마지막 좌/우 facing을 사용한다

### 5-4. 공격 판정 형태

현재 공격은 아래 형태다.

- 크기: `1U x 1U`
- 형태: 회전 가능한 직사각형
- 위치: 플레이어 몸체 앞쪽으로 배치
- 회전: 마우스 방향 각도에 맞춤

즉, 현재 공격은
"마우스 방향으로 회전한 1U 사각형 판정"이다.

### 5-5. 공격 결과

- 범위 안의 낙하 블록을 모두 타격 가능
- 같은 클릭 1회 안에서 같은 블록은 한 번만 맞음
- 블록을 하나도 맞추지 못하면 miss 상태 문구 출력

---

## 6. 채굴 시스템

### 6-1. 기본 원칙

현재 채굴은 아래 성질을 가진다.

- 우클릭 입력
- 공격과 별도 소비
- 모래와 벽 모두 대상
- 플레이어 기준 방향 판정 사용
- 공격보다 좁고 짧은 범위 사용

### 6-2. 현재 수치

| 항목 | 값 |
|---|---:|
| 채굴 피해 | `1` |
| 채굴 거리 | `0.25U` |
| 채굴 높이/두께 | `1U` |
| 채굴 쿨다운 | `0.15초` |
| 채굴 입력 버퍼 | `0.12초` |

### 6-3. 채굴 방향

현재 채굴은 공격과 같은 방향 해석을 재사용한다.
즉, 현재 코드는 좌/우 전용 제한이 아니라
마우스 방향 기반의 좁은 회전 직사각형을 사용한다.

### 6-4. 채굴 처리 순서

현재 채굴 한 번에서 아래가 수행된다.

1. 채굴 형상을 계산한다
2. `SandField`에 모래 채굴을 시도한다
3. `WorldGrid`에 벽 채굴을 시도한다
4. 둘 다 맞지 않으면 miss 상태 문구를 남긴다

즉, 현재는 “모래 우선 후 벽 fallback”만 있는 단순 구조가 아니라,
같은 채굴 액션에서 모래와 벽이 모두 맞을 수도 있다.

### 6-5. 채굴 결과 문구

현재는 다음 성격의 상태 문구를 사용한다.

- Sand hit/remove 결과
- Wall hit/remove 결과
- 아무것도 없으면 `Nothing to mine here.`

---

## 7. 피격과 압착 규칙

현재 피격은 아래 조건에서만 발생한다.

- 낙하 블록이 플레이어와 겹치려 한다
- 플레이어 아래에 지지 요소가 있다
- 즉, 더 이상 아래로 밀려날 수 없는 압착 상태다

지지 요소는 아래를 포함한다.

- 고정 지형(벽/바닥)
- 모래
- 다른 낙하 블록

피격 결과:

- 체력 1 감소
- 무적 시간 `0.5초`
- 해당 블록은 분해 처리
- 상태 텍스트 갱신

단순 측면 접촉이나 스침은 피격이 아니다.

---

## 8. 낙하 블록 시스템

### 8-1. 기본 낙하

현재 낙하 블록은 아래 성질을 가진다.

- 중앙 샤프트 상단 바깥에서 생성
- 매 프레임 1px 단위 누적 낙하
- 낙하 속도 `226 px/s`
- 바닥이나 모래와 닿기 전까지 낙하 지속

### 8-2. 플레이어와의 관계

낙하 중 블록은 아래 관계를 가진다.

- 공격 대상이 된다
- 플레이어 몸체와 겹치면 그대로 통과하지 않는다
- 압착 조건이면 플레이어에게 피해를 준다
- 압착 조건 전이면 플레이어 머리 위에 잠시 멈출 수 있다

### 8-3. 파괴

파괴는 공격으로 체력이 0 이하가 되었을 때 발생한다.

결과:

- 블록 즉시 제거
- 모래 생성 없음
- 골드 지급
- 데미지 팝업 표시

### 8-4. 분해

분해는 아래 경우에 발생한다.

- 블록이 바닥에 닿음
- 블록이 이미 쌓인 모래에 닿음
- 블록이 플레이어를 압착해 피격을 발생시킴

결과:

- 블록 제거
- 해당 블록 데이터 기준 모래 생성
- 골드 없음

현재 구현에서 분해 이유는 `settled`와 `player_crush`로 구분된다.

---

## 9. 블록 데이터 구조

현재 블록은 단순 3종 구조가 아니라,
블록 타입과 블록 베이스를 함께 가진 데이터 구조를 사용한다.

### 9-1. 색상 키

현재 색상 키는 아래 3종이다.

- `amber`
- `cobalt`
- `ember`

각 색상 키는 블록 색 / 모래 색 / 모래 무게를 결정한다.

### 9-2. 블록 베이스

현재 블록 베이스는 아래 8종이 정의돼 있다.

- glass
- wood
- rock
- marble
- gold
- cement
- steel
- bomb

이 데이터는 아래에 사용된다.

- 표시 색상
- HP 배율
- 스폰 가중치
- 보상 배율
- 특수 결과 타입

### 9-3. 현재 구현 주의점

블록 베이스 데이터와 특수 결과 타입은 이미 테이블에 존재하지만,
유리/황금/폭탄의 특수 결과 연출과 판정은 아직 전체 런타임 흐름에 완전히 연결된 단계로 보지 않는다.
즉, 현재는 데이터 구조가 먼저 깔려 있고,
특수 결과의 본격적인 체감 확장은 후속 작업 범위다.

---

## 10. 모래 시스템

### 10-1. 모래 생성

모래는 분해된 블록에서만 생성된다.

현재 생성 방식은 블록 직사각형을 그대로 변환하는 완전 부피 보존 시뮬레이션이 아니라,
블록 데이터의 `sand_units` 수만큼의 셀을 하단 쪽부터 채워 넣는 방식이다.

즉, 현재는
"블록 크기에 비례한 누적 압박"
을 우선한다.

### 10-2. 모래 셀 구조

| 항목 | 값 |
|---|---:|
| 1U 당 모래 셀 수 | `6` |
| 모래 셀 크기 | `64 / 6 ≈ 10.67px` |

모래는 블록보다 더 잘게 쪼개진 셀 기반 물질이다.

### 10-3. 모래 흐름

현재 모래 흐름은 아래 순서를 따른다.

1. 아래 칸으로 낙하 시도
2. 아래가 막히면 좌하/우하 대각선 시도
3. 좌우 우선순위는 틱마다 뒤집음

### 10-4. 활성 영역 제한

모든 모래를 매 프레임 다 갱신하지 않는다.

현재는:

- 플레이어 주변을 활성 영역으로 표시
- 활성 셀만 업데이트
- 틱당 업데이트 수에 상한 존재

이 구조는 성능 안정성의 핵심 제약이다.

---

## 11. 모래와 플레이어 상호작용

### 11-1. 모래 위 발판

- 모래는 플레이어를 지지할 수 있다
- 플레이어는 모래 위에서 점프할 수 있다

### 11-2. 모래 위 이동 저항

- 모래 위에서는 이동 속도 배율 `0.62` 적용
- 일반 지형보다 느리게 움직인다

### 11-3. 모래 밀기

플레이어가 수평 이동 중 모래와 충돌하면,
근처 모래를 연쇄적으로 밀어내는 시도를 한다.

현재 구현 특징:

- 전체 더미를 통째로 미는 방식이 아니다
- 후보 수 / 검사 수 / 이동 수에 상한이 있다
- 막혔던 밀기 시도는 서명 캐시로 반복 계산을 줄인다

### 11-4. 점프 공간 확보

플레이어가 위쪽으로 이동하려다 모래와 충돌하면,
점프 공간 확보 시도를 한다.

현재 구현 특징:

- 후보 수 제한
- 검사 수 제한
- 이동 수 제한
- 실패 시 재시도 프레임 지연

즉, 현재 코드 기준으로 점프 공간 확보는 아직 살아 있는 보조 규칙이다.

---

## 12. 모래 채굴 규칙

### 12-1. 현재 원칙

현재 모래는 공격으로는 제거되지 않지만,
채굴로는 제거 가능하다.

### 12-2. 현재 구현 방식

현재 `SandField`는 두 계층의 모래 채굴 인터페이스를 가진다.

- shape 기반 채굴
- rect 기반 채굴

현재 런타임 메인 루프에서는 shape 기반 채굴이 사용된다.

### 12-3. 채굴 결과

- 채굴 범위와 겹친 모래 셀은 HP가 감소한다
- HP가 0이 되면 셀이 제거된다
- 삭제 직후 전체 더미를 즉시 재정렬하지 않는다
- 삭제 셀 주변만 active 처리하고, 이후 여러 프레임에 걸쳐 다시 흐른다

즉, 현재 모래 채굴은
"즉시 전체 재배치"가 아니라
"국소 삭제 후 점진적 안정화"
를 따른다.

---

## 13. 낙하 블록 위 탑승

현재 구현에서 플레이어는 낙하 블록 위에 올라탈 수 있다.

구체적으로는 아래 동작을 가진다.

- 발밑 지원 블록을 감지한다
- 블록이 이번 프레임에 얼마나 움직였는지 추적한다
- 플레이어는 그 이동량만큼 같이 이동한다
- 필요할 때 블록 상단에 스냅되어 들썩임을 줄인다

즉, 현재 낙하 블록은
고정 발판은 아니지만,
부분적으로 이동 플랫폼처럼 작동한다.

## 14. 런타임 HUD 및 상태 정보

현재 플레이와 직접 연결되는 런타임 정보는 아래와 같다.

- 골드
- 체력
- 상태 문구
- 디버그 라인(`Blocks | Sand | Wall`)
- 대시 디버그 텍스트
- 마지막 스폰 블록 디버그 텍스트

즉, 현재 HUD는 단순 표시를 넘어서
현재 루프 검증에 필요한 디버그 정보까지 포함한다.

## 15. 테스트 기준

현재 시스템이 유지된다고 보기 위한 최소 기준은 아래와 같다.

- 좌우 이동, 더블 점프, 벽 점프, 빠른 낙하가 동작해야 한다
- 좌 / 우 / 하 대시가 더블탭 기반으로 동작해야 한다
- 마우스 방향 360도 공격이 가능해야 한다
- 범위 안 여러 낙하 블록 동시 타격이 가능해야 한다
- 우클릭 채굴로 모래와 벽이 반응해야 한다
- 블록 파괴 시 골드가 지급되어야 한다
- 블록 분해 시 모래가 남아야 한다
- 모래가 아래/대각선으로 자연스럽게 적층되어야 한다
- 플레이어가 모래를 밀며 이동할 수 있어야 한다
- 플레이어가 모래 위에서 점프할 수 있어야 한다
- 플레이어가 낙하 블록 위에 올라탄 상태에서 점프 타이밍이 지나치게 흔들리지 않아야 한다
- 압착 조건에서만 피격이 발생해야 한다

## 16. 결론

현재 Burial Protocol의 게임플레이 시스템은
즉시 위협인 낙하 블록과,
실패 결과로 남는 누적 환경인 모래를,
공격 / 채굴 / 점프 / 대시 / 위치 선정으로 동시에 관리하게 만드는 구조다.

이 문서의 핵심은
플레이 중 벌어지는 규칙을 플레이어, 블록, 모래로 분산하지 않고
한 기준 문서 안에 묶어 유지하는 데 있다.
