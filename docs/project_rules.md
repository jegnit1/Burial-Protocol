# Burial Protocol - Project Rules

## 0. 목적

이 문서는 Burial Protocol 개발의 최상위 작업 기준을 정리한다.
현재 구현된 프로토타입의 구조, 문서 우선순위, 작업 범위, 코드 작성 규칙, 테스트 원칙을 고정해
이후 작업이 같은 기준 위에서 누적되도록 만드는 것이 목적이다.

이 문서는 아래 역할을 가진다.

- 현재 프로젝트의 공통 전제를 고정한다
- 문서 간 해석 충돌 시 우선순위를 제공한다
- 현재 프로토타입 범위와 비범위를 구분한다
- 코드/씬/데이터 분리 원칙을 정한다
- 문서와 구현의 싱크 기준을 명확히 한다

---

## 1. 프로젝트 기본 정보

- 프로젝트명: Burial Protocol
- 엔진: Godot 4.6
- 언어: GDScript
- 플랫폼: PC
- 화면 방향: 세로형 플레이 중심
- 장르 구조: 2D 액션 생존 + 낙하 블록 압박 + 누적 모래 환경

현재 구현 스냅샷의 핵심 수치는 아래와 같다.

| 항목 | 현재 값 |
|---|---|
| 뷰포트 | 1920 x 1664 |
| 1U / CELL_SIZE | 64px |
| 중앙 샤프트 폭 | 10칸 |
| 좌우 벽 두께 | 각 10칸 |
| 월드 총 너비 | 30칸 |
| 월드 총 높이 | 26칸 |

프로젝트의 핵심 감각은 아래와 같다.

- 상단에서 낙하 블록이 계속 등장한다
- 플레이어는 마우스 방향으로 360도 공격할 수 있다
- 기본 액션 입력은 블록이 없으면 좌우 벽 채굴로 이어진다
- 제거하지 못한 블록은 모래가 되어 누적된다
- 누적된 모래는 이동 저항이면서 발판 역할도 한다
- 플레이어는 모래를 밀고, 오르고, 벽을 파며 생존 공간을 유지한다

## 2. 문서 우선순위

문서 해석 충돌이 발생하면 아래 순서를 따른다.

1. `project_rules.md`
2. `core_loop_spec.md`
3. `block_and_sand_system_spec.md`
4. `player_and_input_spec.md`
5. `scaling_change_note.md`

원칙은 아래와 같다.

- 상위 문서가 하위 문서보다 우선한다
- 하위 문서가 오래되었으면 구현이 아니라 문서를 수정 대상으로 본다
- 현재 구현과 문서가 어긋난 것이 확인되면 문서를 즉시 동기화한다
- `GameConstants.gd`, `Main.gd`, `Player.gd`, `FallingBlock.gd`, `SandField.gd`는 현재 구현 검증의 핵심 파일이다

## 3. 현재 작업 범위

현재 프로젝트는 플레이 가능한 핵심 루프를 다듬는 단계다.
현재 범위에 포함되는 항목은 아래와 같다.

- 플레이어 이동, 점프, 벽 점프, 빠른 낙하
- 낙하 블록 생성, 공격, 파괴, 분해
- 360도 공격 판정과 시각화
- 좌우 벽 채굴
- 모래 생성, 적층, 흐름, 밀기, 점프 공간 확보
- HUD, 상태 텍스트, 디버그 패널
- 낙하 블록 위 탑승 및 이동 플랫폼 보정

현재 범위 밖 항목은 아래와 같다.

- 상점, 성장, 유물, 상인
- 스테이지 구조, 저장, 메타 진행
- 별도 게임 오버 씬/연출
- 특수 공격, 특수 모래 제거 수단
- 온라인 기능, 외부 서비스 연동

주의할 점은 아래와 같다.

- 현재는 체력이 0이 되어도 별도 종료 씬으로 넘어가지 않는다
- 수동 재시작은 `R`로 처리한다
- 골드는 파괴 피드백 수단으로만 존재하며 소비처는 없다

---

## 4. 구현 우선순위

작업 우선순위는 아래 순서를 따른다.

1. 실제 플레이 가능 여부
2. 규칙의 일관성
3. 입력/판정의 안정성
4. 문서와 구현의 일치
5. 데이터 분리
6. 시각적 완성도

즉, 현재 단계에서는
"예쁘게 보이는 것"보다
"반복 플레이가 안정적으로 성립하는 것"이 먼저다.

---

## 5. 시스템 설계 원칙

### 5-1. 역할 분리

핵심 역할은 아래처럼 분리한다.

- `Main`: 전체 연결, 스폰, 공격/채굴 분기, 카메라 설정
- `Player`: 이동, 점프, 공격 방향 해석, 공격 미리보기, 낙하 블록 탑승 보정
- `FallingBlock`: 낙하, 피격 처리, 파괴/분해 신호, 데미지 팝업 생성
- `SandField`: 모래 셀 생성, 흐름, 밀기, 점프 공간 확보
- `WorldGrid`: 고정 벽/바닥, 월드 좌표 변환, 채굴 처리
- `HUD`: 골드, 체력, 상태 문구, 디버그 수치 표시
- `GameConstants`: 공용 수치와 데이터 정의
- `GameState`: HUD용 런타임 상태 저장

### 5-2. 규칙 기반 처리 우선

- 낙하 블록과 모래는 엔진 물리에 전부 위임하지 않는다
- 플레이어 이동, 모래 흐름, 블록 낙하는 규칙 기반으로 유지한다
- 결과가 흔들리는 자동 물리보다 예측 가능한 판정을 우선한다

### 5-3. 현재 구현 감각 우선

새 기능을 넣을 때는 아래 감각을 깨지 않는지 먼저 본다.

- 블록은 즉시 위협이어야 한다
- 공격은 가장 효율적인 공간 확보 수단이어야 한다
- 실패한 블록은 모래 누적으로 되돌아와야 한다
- 모래는 위험이면서 이동 자산이어야 한다
- 채굴은 좌우 생존 공간을 다시 여는 행동이어야 한다

---

## 6. 수치 관리 원칙

현재 프로젝트의 핵심 수치는 `GameConstants.gd`에 모은다.

특히 아래 값은 코드 내부 매직 넘버로 흩어놓지 않는다.

- 월드/뷰포트 크기
- 플레이어 이동/점프/피격 수치
- 공격 범위와 공격 쿨다운
- 블록 낙하 속도와 스폰 간격
- 모래 시뮬레이션 분해능과 예산
- HUD 크기, 폰트 크기
- 데미지 팝업 연출 값
- 블록 타입 데이터

블록 타입처럼 구조화된 값은 데이터 테이블 형태를 유지한다.

---

## 7. 코드 작성 규칙

### 7-1. 일반 규칙

- 현재 작업과 직접 관련된 파일만 수정한다
- 무관한 대규모 리팩토링은 하지 않는다
- 책임이 다른 시스템을 섞지 않는다
- 사용하지 않을 범용 구조를 미리 만들지 않는다

### 7-2. 주석 규칙

- 주석은 한국어로 작성한다
- 값의 역할, 판정 의도, 계산 목적이 바로 드러나야 한다
- 너무 긴 설계 토론은 코드가 아니라 문서에 남긴다

### 7-3. 네이밍 규칙

- 씬/스크립트 파일: PascalCase
- 함수/변수/신호/데이터 키: snake_case
- 상수: UPPER_SNAKE_CASE

### 7-4. 하드코딩 제한

- 값이 반복되거나 밸런스 대상이면 상수화한다
- 일회성 로컬 계산은 허용하지만, 시스템 수치는 상수화한다

---

## 8. 씬 및 데이터 분리 원칙

### 8-1. 한 씬은 한 책임

각 씬은 자기 책임을 벗어난 로직을 최대한 가져가지 않는다.

### 8-2. 데이터 우선 구조

아래 항목은 데이터/상수 우선으로 관리한다.

- 블록 타입별 크기, 체력, 보상, 모래량, 색상
- 모래 색상별 무게
- 공격 관련 수치
- HUD/뷰포트/월드 배치 값

### 8-3. 현재 허용되는 직접 참조

현재 구조에서는 아래 정도의 직접 참조를 허용한다.

- `Main`이 `Player`, `SandField`, `WorldGrid`, `HUD`를 연결
- `Player`가 `blocks_root`를 받아 발밑 블록 확인
- `FallingBlock`이 플레이어 압착 여부를 질의

단, 다른 씬의 내부 노드 구조에 깊게 의존하는 방식은 계속 피한다.

---

## 9. 테스트 원칙

### 9-1. 작업 후 확인 필수 항목

작업 후 최소 아래를 확인한다.

- 헤드리스 로드가 되는가
- 메인 씬이 깨지지 않는가
- 핵심 입력이 여전히 동작하는가
- 기존 코어 루프가 끊기지 않는가

### 9-2. 현재 우선 테스트 항목

- 블록 공격과 다중 타격
- 좌우 벽 채굴
- 블록 파괴/분해 결과 차이
- 모래 생성과 흐름
- 모래 위 이동과 점프
- 낙하 블록 위 탑승과 점프 안정성
- 골드/체력/HUD 갱신

---

## 10. 금지 사항

아래 작업은 금지한다.

- 문서 근거 없는 기능 임의 도입
- 코어 루프와 무관한 대규모 리팩토링
- 블록/모래 전체를 엔진 물리에 위임하는 구조
- 개별 모래 입자를 대량 RigidBody로 만드는 구조
- 설명 없는 매직 넘버 추가
- 외부 플러그인 의존성 추가
- 미구현 시스템을 문서만 믿고 구현된 것처럼 서술

---

## 11. 허용되는 임시 구현

현재 단계에서 아래는 허용된다.

- 도형 기반 플레이스홀더 비주얼
- 텍스트 기반 상태 피드백
- 디버그 패널
- 단순 데미지 팝업
- 임시 밸런스 수치

단, 임시 구현도 아래를 만족해야 한다.

- 현재 루프를 더 잘 검증하게 도와야 한다
- 이후 교체하기 쉬워야 한다
- 구조를 어지럽히지 않아야 한다

---

## 12. 문서 동기화 원칙

문서는 구현이 바뀔 때 같이 바뀌어야 한다.
특히 아래 변화가 생기면 문서를 반드시 갱신한다.

- 입력 체계 변경
- 공격 방향/판정 구조 변경
- 블록 파괴/분해 규칙 변경
- 모래 이동 규칙 변경
- 월드 크기/스케일 변경
- 런 상태와 HUD 표시 변경

현재 문서 세트는 "미래 희망 스펙"이 아니라
"현재 구현과 다음 작업의 기준 문서"를 목표로 유지한다.

---

## 13. 결론

Burial Protocol의 현재 개발 목표는
완벽한 콘텐츠 확장이 아니라,
낙하 블록, 모래 누적, 공격, 채굴, 점프, 생존 공간 관리가
실제로 반복 가능한 상태를 만드는 것이다.

따라서 항상 아래를 우선한다.

- 문서와 구현의 일치
- 반복 플레이의 안정성
- 입력과 판정의 명확성
- 코어 루프를 해치지 않는 확장
